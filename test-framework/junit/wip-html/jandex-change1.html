The Core Problem

During JUnit test discovery, FacadeClassLoader.loadClass() is called for every class JUnit encounters. For each class, the FacadeClassLoader must determine: Is this a @QuarkusTest? If yes, route it to a QuarkusClassLoader; if no, delegate to the parent (system) classloader.

The question: How do you inspect a class's annotations to determine if it's a @QuarkusTest, without triggering the heavy side-effects of actually loading the class?

What is Jandex Indexing?

The Jandex Indexer reads .class files as raw bytecode and parses out structural information (class names, superclass, interfaces, annotations, fields, methods) without loading them into the JVM. The result is an in-memory Index — a set of hash maps keyed by class name and annotation name.

Indexer indexer = new Indexer(); for (String spec : classesPath.split(File.pathSeparator)) { Path path = Path.of(spec); if (Files.isDirectory(path)) { // Only directories, not JARs indexDirectory(indexer, path); // Walk tree, index each .class file } } jandexIndex = indexer.complete(); // Build immutable Index
Why only directories? Test classes always live in directories (target/test-classes, target/classes), not in JARs. Classes from library JARs are never user test classes, so they don't need to be indexed — they get the fast path (super.loadClass()) in loadClass().

Continuous Testing Path

In continuous testing mode, the set of QuarkusTest classes is pre-computed by JunitTestRunner and passed into the constructor. The key change here is simpler:

Before
if (isContinuousTesting) { isQuarkusTest = quarkusTestClasses.contains(name); profile = profiles.get(name); // Class<?> if (isQuarkusTest) { inspectionClass = peekingClassLoader.loadClass(name); // Still loaded class for inspection! } }
After
if (isContinuousTesting) { isQuarkusTest = quarkusTestClasses.contains(name); profileClassName = profileNames != null ? profileNames.get(name) : null; // No class loading at all! }
Annotation Detection Methods

Each annotation check was reimplemented to work with Jandex's ClassInfo instead of loaded Class<?> objects.

1. isQuarkusTest()
Before
private boolean isQuarkusTest(Class<?> inspectionClass) { boolean isQuarkusAtTopLevel = AnnotationSupport.isAnnotated( inspectionClass, quarkusTestAnnotation) || registersQuarkusTestExtensionWithExtendsWith(inspectionClass) || registersQuarkusTestExtensionOnField(inspectionClass); if (isQuarkusAtTopLevel) return true; // Nested class check boolean isNested = AnnotationSupport.isAnnotated( inspectionClass, nestedAnnotation); if (isNested) { String parentName = inspectionClass.getName() .substring(0, inspectionClass.getName().lastIndexOf("$")); Class<?> parent = peekingClassLoader.loadClass(parentName); return isQuarkusTest(parent); } return false; }
After
private boolean isQuarkusTest(ClassInfo classInfo) { if (hasQuarkusTestMarker(classInfo)) return true; // Walk superclass hierarchy (NEW!) DotName superName = classInfo.superName(); while (superName != null && !OBJECT_NAME.equals(superName)) { ClassInfo superInfo = jandexIndex.getClassByName(superName); if (superInfo != null) { if (hasQuarkusTestMarker(superInfo)) return true; superName = superInfo.superName(); } else { // Not in index → reflection fallback if (isQuarkusTestViaReflection(superName.toString())) return true; break; } } // Nested class check via enclosingClass() if (classInfo.hasDeclaredAnnotation(NESTED)) { DotName enc = classInfo.enclosingClass(); // No string manipulation, no class loading ... } }
2. Meta-Annotation Walking
Before

JUnit's AnnotationSupport.isAnnotated() handles meta-annotations automatically by walking the annotation hierarchy using reflection on loaded Class<?> objects.

// Single call handles meta-annotations AnnotationSupport.isAnnotated( inspectionClass, quarkusTestAnnotation); // But requires the class AND annotation // to be loaded in the same classloader
After

Custom recursive meta-annotation walking within the Jandex index. Only annotations that are themselves in the index are walked.

private boolean hasAnnotation( ClassInfo classInfo, DotName target) { if (classInfo.hasDeclaredAnnotation(target)) return true; // Walk meta-annotations within index for (AnnotationInstance ann : classInfo.declaredAnnotations()) { ClassInfo annCI = jandexIndex.getClassByName(ann.name()); if (annCI != null && hasAnnotationRecursive(annCI, target, new HashSet<>())) return true; } return false; }
3. @DisabledOnOs Check
Before
private boolean isDisabledOnOs(Class<?> inspectionClass) { Optional<?> ma = AnnotationSupport.findAnnotation( inspectionClass, disabledOnOsAnnotation); if (ma.isPresent()) { Object values = disabledOnOsAnnotationValue.invoke(a); // Array reflection because OS enum is // in a different classloader int length = Array.getLength(values); for (int i = 0; i < length; i++) { Object value = Array.get(values, i); boolean matches = (boolean) osIsCurrent.invoke(value); ... } } }
After
private boolean isDisabledOnCurrentOs(ClassInfo classInfo) { AnnotationInstance ann = classInfo.declaredAnnotation(DISABLED_ON_OS); if (ann == null) return false; AnnotationValue value = ann.value(); if (value == null) return false; for (String osName : value.asEnumArray()) { if (OS.valueOf(osName).isCurrentOs()) return true; } return false; } // No reflection, no Array manipulation, // no cross-classloader enum issues
4. @ExtendWith & @RegisterExtension
Before
// @ExtendWith - string matching on toString() Optional<?> a = AnnotationSupport.findAnnotation( inspectionClass, extendWithAnnotation); return a.isPresent() && a.get().toString().contains( QuarkusTestExtension.class.getName()); // @RegisterExtension - field scanning via reflection List<Field> fields = AnnotationSupport .findAnnotatedFields(inspectionClass, registerExtensionAnnotation, f -> f.getType().getName().equals(...)); // Sometimes throws NoClassDefFoundError!
After
// @ExtendWith - type-safe class array check AnnotationInstance ann = classInfo.declaredAnnotation(EXTEND_WITH); for (Type type : ann.value().asClassArray()) { if (type.name().toString() .equals(QUARKUS_TEST_EXTENSION_NAME)) return true; } // @RegisterExtension - Jandex field info for (FieldInfo field : classInfo.fields()) { if (field.hasAnnotation(REGISTER_EXTENSION) && field.type().name().toString() .equals(QUARKUS_TEST_EXTENSION_NAME)) return true; } // No NoClassDefFoundError possible!

Why load from parent CL? Downstream APIs (TestResourceUtil.getReloadGroupIdentifier(), TestProfileAndProperties.of(), AppMakerHelper) require Class<?> objects. By loading from the parent (system) CL instead of peekingCL, we avoid the second parallel classloader and load only the specific test/profile classes that are actually QuarkusTests — not every discovered class.

Impact & Tradeoffs
Benefits
Category Before (Problem) After (Solution)
Performance Every class loaded twice: once by peekingCL for inspection, once by runtimeCL for execution. URLClassLoader overhead for each. Annotation inspection via O(1) index lookup. Only QuarkusTest classes loaded (once). Non-test classes get immediate delegation.
Stability NoClassDefFoundError when peekingCL loads classes with missing transitive dependencies. Required try/catch workarounds. Jandex reads raw bytecode — no dependency resolution. Missing classes simply don't appear in the index.
Side effects Static initializers run during peeking load. Class-level side effects (resource loading, registration) triggered prematurely. No class loading = no JVM class initialization = no side effects during inspection.
Cross-CL issues Annotations loaded by peekingCL incompatible with parent CL's versions. Required reflective getMethod("value").invoke() and Array.get() for enum values. DotName constants are plain strings — no classloader boundary issues. Enum values read directly as strings from bytecode.
Code complexity ~13 mutable fields (Class<?> + Method refs). Special handling for isolated vs launcher classloaders. ParentLastURLClassLoader workaround class. 1 Index field + 1 boolean + 10 DotName constants (static final). ParentLastURLClassLoader deleted.
Memory URLClassLoader holds open file handles to all classpath entries. All classes loaded into metaspace by peekingCL (eventually GC'd). Jandex Index is a pure data structure — no file handles, no metaspace pressure. Nullified on close() for immediate GC.
Tradeoffs & Edge Cases
Consideration Details
Index build time Jandex walks target/test-classes and target/classes directories at construction time, reading every .class file. For large projects this is fast (Jandex is optimized for this), but it's a new upfront cost that replaces the per-class cost of peekingCL. Net effect is typically positive because the upfront scan is cheaper than loading hundreds of classes.
Meta-annotations from JARs Custom annotations defined in library JARs (not in test directories) that are themselves annotated with @QuarkusTest won't be found via Jandex meta-annotation walking, because the annotation class isn't in the index. This is a very rare edge case — the isQuarkusTestViaReflection() fallback handles the superclass case but not the JAR-defined meta-annotation case.
JARs on classpath Only directories are indexed, not JARs. This is by design — test classes always live in directories. But if someone had test classes inside a JAR on the classpath, they wouldn't be discovered as QuarkusTests. This matches previous behavior since peekingCL also primarily targeted directory-based classes.
Jandex dependency Jandex (io.smallrye:jandex) is already a transitive dependency of the test framework module. No new dependency added. It's also heavily used throughout the Quarkus codebase for build-time indexing.
